# Report Generation / 智能报告生成

This module builds evidence-grounded road-damage reports from the detection, video deduplication, and area-estimation modules.

本模块基于检测、视频去重和面积估计模块的结构化证据生成道路病害报告。

## Position / 模块位置

The final runnable path is:

最终可运行链路是：

```text
02_detection -> 03_video_dedup -> 04_area_measurement -> 05_report_generation
```

`01_segmentation` remains an exploration/display module and is not required for report generation.

`01_segmentation` 保留为探索/展示模块，不是报告生成的必要输入。

## Core Design / 核心设计

Do not ask a vision-language model to infer everything from the image or video directly. The model should receive structured evidence generated by the upstream tools.

不要让视觉语言模型直接看图/视频自由发挥。报告生成必须以结构化证据为输入。

The input should include:

输入应包含：

- detection summary: class counts, confidence, representative boxes;
- video dedup summary: unique events, first/last timestamp, keyframes;
- area summary: estimated area by method and method limitations;
- visual evidence: selected raw frames, annotated frames, area boards;
- system limitations: no physical area GT, no camera calibration, confidence is not accuracy.

## Recommended First Version / 第一版建议

Use prompt-based report generation first. Do not fine-tune Qwen-VL yet.

第一版先做提示词工程，不建议立刻微调 Qwen-VL。

Reasons:

原因：

- DamageQwen shows that structured detector outputs improve VLM descriptions.
- Your current pipeline already produces structured evidence.
- You do not yet have enough expert-written road inspection reports for reliable LoRA fine-tuning.
- The thesis needs controllable, verifiable reports more than fluent but unsupported text.

## Files / 文件

- `docs/agent_design.md`: research mapping and final agent architecture.
- `schemas/report_input.schema.json`: proposed report evidence schema.
- `prompts/report_writer.md`: evidence-grounded report writer prompt.
- `prompts/report_verifier.md`: verifier prompt for hallucination and unsupported claims.
- `scripts/report_pipeline.py`: builds image/video report evidence and optionally calls SiliconFlow Qwen.

## Usage / 使用

Set the SiliconFlow API key in your shell. Do not write it into any file.

先在 shell 中设置 SiliconFlow API key。不要写入任何文件。

```bash
export SILICONFLOW_API_KEY="your_api_key_here"
```

Build an image report demo:

生成图片报告 demo：

```bash
python road_damage_pipeline/05_report_generation/scripts/report_pipeline.py \
  --mode image \
  --call-api
```

Build a video report demo:

生成视频报告 demo：

```bash
python road_damage_pipeline/05_report_generation/scripts/report_pipeline.py \
  --mode video \
  --call-api
```

Dry-run without calling the API:

只生成证据和请求预览，不调用 API：

```bash
python road_damage_pipeline/05_report_generation/scripts/report_pipeline.py --mode image
python road_damage_pipeline/05_report_generation/scripts/report_pipeline.py --mode video
```

## Outputs / 输出

Each demo writes:

每个 demo 输出：

- `report_input.json`: full structured evidence.
- `qwen_request_preview.json`: request preview with base64 images removed.
- `raw_response.json`: SiliconFlow response when `--call-api` is used.
- `report.md`: generated report or dry-run placeholder.
- `predicted_images/` or `representative_frames/`: model-predicted visual evidence.

## Boundary / 边界

- The agent may summarize evidence, but must not invent new detections.
- The agent may compare estimated area methods, but must not call them true physical GT.
- The agent may use confidence scores, but must not call confidence "accuracy".
- The agent may give maintenance suggestions, but must mark them as decision support, not engineering diagnosis.
- The agent receives model-predicted boxes only. Ground-truth labels are never sent to Qwen.
- The formal report area input uses M1/M3/M4. FastSAM is kept as exploration evidence only.
